## Problem-definition framework

Define four elements before going broad on analysis:

- Opening Scene or Starting Point
- Disturbing Event
- R1: Undesired Result
- R2: Desired Result

Then identify the reader's question.

### Opening Scene

State the stable structure or process in which the problem lives.

Keep it simple enough to visualize.

### Disturbing Event

State what changed, or what became visible, that disturbed the prior state.

Common categories:

- external change
- internal change
- newly recognized evidence of a problem or opportunity

### R1

State the result you do not want.

### R2

State the result you want instead.

Make R2 as specific as possible. Quantify it when the task allows.

## Common reader positions

Most problem-solving documents answer one of these:

1. We do not know how to get from R1 to R2.
2. We think we know the solution, but are not sure it is right.
3. We know the solution, but need the implementation path.
4. We tried a solution and it failed.
5. We have multiple alternatives and need to choose.
6. We know change is needed but not the target state.
7. We know the target state but are not sure a problem exists.

Use the reader's actual position to choose the structure.

## Diagnostic thinking before data

Avoid this pattern:

- gather all available data
- label it under generic categories
- hope conclusions emerge later

Prefer this pattern:

- define the problem
- generate likely explanations
- create diagnostic frameworks or logic trees
- gather only the evidence needed to confirm or reject them

Use mutually exclusive, collectively exhaustive breakdowns when exploring causes or options.

## Logic trees

Use logic trees to:

- explore possible causes
- generate alternative actions
- check completeness
- reveal flaws in grouped ideas

Build the tree so each branch reflects a real analytical choice, not a random topic list.

## Issue analysis

Prefer crisp, decision-driving issues.

A true issue is usually phrased as a yes-or-no question.

Good:

- Is the current inventory level too high?
- Is the centralized ordering system placing orders properly?
- Should we reorganize functionally?

Weak:

- What are the key issues?
- How should we reorganize?

Those may be prompts, but they do not define a clean decision point.

If an "issues" section is really a process, recommendation, or analysis tree, name it accordingly.

## Presentation rules

Do not treat a presentation as report text on slides.

Use the pyramid to determine what belongs on screen and what should stay in speech.

### Storyboarding workflow

1. Write the introduction in full.
2. Put the introduction points, Key Line points, and one level below onto a blank storyboard.
3. Sketch how each point should be shown visually.
4. Script the words around the slides.
5. Finish slide design.
6. Rehearse until the flow is natural.

### Slide rules

- Give each slide a sentence or phrase that states its point.
- Use text slides to clarify structure.
- Use exhibit slides to prove a point visually.
- Keep the deck aligned with the spoken story.

### Next Steps sections

Use a `Next Steps` section only for actions that are obvious once the recommendation is accepted.

If a proposed next step still needs proof, argument, or defense, move it back into the body of the document.
